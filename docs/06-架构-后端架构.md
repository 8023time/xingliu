# 后端：应用架构与实现设计

## 1. 后端目标

`server` 是星流平台的统一 API 服务，负责同时支撑：

- B 端创作者后台：登录注册、内容草稿、版本管理、素材、Prompt、AI 创作、审核、评分、改写、发布和分发管理。
- C 端内容前台：已发布内容详情、信息流、榜单、阅读互动和数据反馈。

当前后端采用 NestJS 模块化单体架构。模块化单体适合训练营阶段快速迭代：代码仍在一个服务内，但按业务域拆分模块，避免把用户、内容、AI、审核、榜单逻辑混在一起。后续如果某些能力压力变大，例如 AI Gateway 或榜单计算，可以再拆成独立服务。

## 2. 技术栈

| 能力 | 当前选型 | 说明 |
| --- | --- | --- |
| HTTP 服务 | NestJS 11 | 模块、Controller、Service、Guard、Interceptor、Filter |
| 语言 | TypeScript | 后端和共享类型统一使用 TS |
| ORM | Prisma 7 | 数据模型、迁移、类型安全查询 |
| 数据库 | PostgreSQL | 主业务库，承载用户、内容、版本、审核、评分、榜单数据 |
| 认证 | JWT | access token / refresh token 双令牌 |
| API 文档 | Swagger + Scalar | `/api-docs` 和 `/docs` |
| 配置 | `@nestjs/config` + `@xingliu/config` | 环境变量和端口配置 |
| 前后端共享类型 | `@xingliu/shared` | 用户、内容等请求/响应类型复用 |

## 4. 启动入口

后端入口是 `server/src/main.ts`。

启动过程：

```text
NestFactory.create(AppModule)
-> 注册全局响应拦截器
-> 注册全局异常过滤器
-> 设置全局前缀 /api
-> 开启 CORS
-> 生成 Swagger 文档
-> 挂载 Scalar API Reference
-> 监听 @xingliu/config 中的 server 端口
```

当前公开入口：

| 路径 | 说明 |
| --- | --- |
| `/api/*` | 业务 API |
| `/api-docs` | Swagger UI |
| `/docs` | Scalar API Reference |

## 6. 公共能力

### 6.1 CommonModule

`CommonModule` 使用 `@Global()` 注册为全局模块，避免每个业务模块重复导入基础能力。

当前导出：

- `PrismaModule`
- `ResponseModule`
- `JwtModule`
- `ConfigModule`
- `CommonService`

### 6.2 PrismaService

`PrismaService` 继承 Prisma 生成的 `PrismaClient`，通过 `@prisma/adapter-pg` 连接 PostgreSQL。

```text
DATABASE_URL
-> PrismaPg adapter
-> PrismaClient
-> 业务 Service 调用 prismaService.xxx
```

Prisma 生成客户端位置：

```text
server/libs/common/src/generated/prisma
```

所有业务模块都应通过 `PrismaService` 访问数据库，不直接创建新的 PrismaClient。

### 6.3 ResponseService

`ResponseService` 用于业务成功/失败返回：

```ts
success(data, message, code = 0)
error(data, message, code = 1)
```

返回结果再经过全局 `HttpResponseInterceptor` 包装成统一结构：

```json
{
  "timestamp": "2026-06-01T10:00:00Z",
  "path": "/api/contents",
  "message": "创建内容成功",
  "success": true,
  "code": 0,
  "data": {}
}
```

### 6.4 全局异常过滤器

`HttpExceptionFilter` 负责处理 Nest 抛出的 HTTP 异常，例如 401、403、404、500。

当前异常结构包含：

- `timestamp`
- `path`
- `message`
- `code`
- `success`

实现时需要注意保持字段名一致，避免响应里出现拼写错误字段。

### 6.5 AuthGuard

`AuthGuard` 从请求头读取：

```text
Authorization: Bearer {accessToken}
```

校验流程：

```text
读取 Authorization
-> 提取 Bearer token
-> JwtService.verify
-> 校验 tokenType 必须是 access
-> 将 payload 写入 req.user
```

后续 Controller 可通过 `req.user.userId` 获得当前用户 ID。

## 7. 认证与用户模块

用户模块当前已经实现：

| 接口 | 说明 |
| --- | --- |
| `POST /api/user/register` | 注册 |
| `POST /api/user/login` | 登录 |
| `POST /api/user/refreshToken` | 刷新令牌 |
| `POST /api/user/logout` | 退出登录 |

认证模型：

```text
accessToken: 15 分钟，用于 API 鉴权
refreshToken: 7 天，用于刷新登录态
tokenVersion: 用户级 token 失效版本号
```

退出登录时后端会校验 refresh token，并递增用户 `tokenVersion`。这样旧 refresh token 会失效。

## 8. 内容模块

`ContentModule` 是当前后端最完整的业务模块，也是后续 AI、审核、评分、发布、榜单的主干。

目录结构：

```text
server/src/content/
├── content.controller.ts
├── content.service.ts
├── content.module.ts
└── dto/
    ├── create-content.dto.ts
    ├── update-content.dto.ts
    ├── content-list-query.dto.ts
    ├── create-content-version.dto.ts
    ├── save-draft-snapshot.dto.ts
    ├── sync-draft-snapshots.dto.ts
    ├── rollback-content.dto.ts
    ├── publish-content.dto.ts
    └── submit-review.dto.ts
```

该模块不使用 `entities`，数据实体以 Prisma schema 为准，前后端数据类型放在：

```text
packages/shared/content/index.ts
```

### 8.1 内容核心模型

内容模块围绕三张主表：

| 表 | 作用 |
| --- | --- |
| `contents` | 当前内容状态，例如标题、当前版本、审核状态、发布状态、分数 |
| `content_versions` | 正式内容版本，每次创建、编辑、改写、回滚都会生成版本 |
| `draft_snapshots` | 编辑过程快照，用于自动保存、手动保存、离线同步和冲突恢复 |

设计思想：

```text
contents 管当前态
content_versions 管正式历史
draft_snapshots 管编辑过程
```

这样 AI 生成、人工编辑、审核、评分、发布都能围绕 `contentId` 串起来。

### 8.2 内容创建流程

接口：

```text
POST /api/contents
```

落库流程：

```text
校验内容类型
-> 校验封面素材归属
-> 创建 contents
-> 创建 content_versions 第 1 版
-> 回写 contents.current_version_id
```

必须使用事务，避免内容主表和版本表状态不一致。

默认状态：

```text
contents.status = DRAFT
contents.safety_status = PENDING
content_versions.change_type = CREATE
```

### 8.3 内容更新流程

接口：

```text
PUT /api/contents/:id
```

更新内容不会直接覆盖正文，而是创建新版本：

```text
读取当前内容和当前版本
-> 合并更新字段
-> 创建新的 content_versions
-> current_version_id 指向新版本
-> 重置审核和评分状态
```

重置状态：

```text
status = DRAFT
safety_status = PENDING
safety_score = null
quality_score = null
quality_level = null
published_at = null
```

原因：内容一旦修改，旧审核结果和旧评分不应继续用于发布。

### 8.4 版本与回滚

版本接口：

| 接口 | 说明 |
| --- | --- |
| `POST /api/contents/:id/versions` | 创建内容新版本 |
| `GET /api/contents/:id/versions` | 查询版本列表 |
| `POST /api/contents/:id/rollback` | 回滚到指定版本 |

回滚不是直接把旧版本设为当前版本，而是复制旧版本内容，生成一个新的 `ROLLBACK` 版本。

价值：

- 历史版本不可变。
- 回滚动作可审计。
- 当前版本永远代表最新操作结果。

### 8.5 草稿快照与离线同步

草稿接口：

| 接口 | 说明 |
| --- | --- |
| `POST /api/contents/:id/draft-snapshots` | 保存草稿快照 |
| `GET /api/contents/:id/draft-snapshots` | 查询草稿快照列表 |
| `GET /api/contents/:id/draft-snapshots/latest` | 查询最新草稿 |
| `POST /api/contents/:id/draft-snapshots/sync` | 离线草稿同步 |

草稿快照不影响正式版本。它服务于编辑器自动保存和断网恢复。

同步冲突判断：

```text
本地 baseServerRevision < 服务端最新 serverRevision
=> 返回 conflict
```

前端收到冲突后，应展示本地版本、云端版本，并允许用户保留本地、采用云端或复制为新草稿。

### 8.6 提交审核与发布

提交审核接口：

```text
POST /api/contents/:id/submit-review
```

当前只负责把内容状态改为 `REVIEWING`，真正审核应由后续 `ModerationModule` 写入 `safety_reviews`。

发布接口：

```text
POST /api/contents/:id/publish
```

发布前检查：

- 内容状态必须是 `APPROVED`。
- 最新安全审核必须是 `PASS`。
- 必须存在质量评分。

发布成功后：

- 更新 `contents.status = PUBLISHED`。
- 初始化或更新 `content_metrics`。
- 写入 `ranking_scores` 的 `LATEST` 记录。
- 写入 `content_operations` 操作审计。

### 8.7 权限边界

所有 B 端内容写操作都必须校验作者归属：

```text
content.id = 请求 id
content.author_id = 当前登录用户
content.deleted_at = null
```

封面素材也需要校验归属：

```text
asset.id = coverAssetId
asset.user_id = 当前登录用户
asset.deleted_at = null
```

## 9. 前后端共享类型

共享类型放在：

```text
packages/shared/
├── user/
└── content/
```

当前内容模块共享：

- 内容类型常量。
- 内容状态常量。
- 草稿保存来源常量。
- 审核模式常量。
- 内容创建、更新、版本、回滚、发布请求类型。
- 内容详情、版本、草稿快照、同步结果响应类型。

后端 DTO 通过 `implements` 共享接口保证请求结构一致，例如：

```ts
export class CreateContentDto implements CreateContentRequest {}
```

前端可直接复用：

```ts
import type { CreateContentRequest, ContentDetailResponse } from '@xingliu/shared/content';
```

## 10. AI Gateway 设计方向

AI 相关模块后续应统一通过 `AiGatewayModule` 调用模型，不允许业务模块直接散落调用 OpenAI。

目标调用链：

```text
业务模块
-> AiGatewayService
-> PromptTemplate / PromptTemplateVersion
-> LangChain
-> OpenAI-compatible 模型
-> 结构化输出校验
-> ai_tasks 审计记录
```

AI Gateway 职责：

- 统一模型配置、baseURL、apiKey、超时。
- 统一 Prompt 版本加载和变量注入。
- 统一 JSON 结构化输出校验。
- 统一记录 `ai_tasks`。
- 统一脱敏、重试和错误处理。

## 11. 数据访问与事务原则

需要事务的场景：

- 创建内容和初始版本。
- 更新内容并创建新版本。
- 回滚内容并写入操作记录。
- 发布内容并初始化指标、榜单记录、操作记录。
- 审核后同时写审核记录并更新内容状态。
- 评分后同时写评分记录、维度明细并更新内容质量分。

事务原则：

- 不让 `contents.current_version_id` 指向不存在的版本。
- 不让发布状态和指标初始化脱节。
- 不让审核/评分记录与内容状态脱节。

## 12. 状态流转

内容主状态：

```text
DRAFT
-> REVIEWING
-> NEED_REWRITE
-> REVIEWING
-> APPROVED
-> PUBLISHED
-> OFFLINE
```

失败分支：

```text
REVIEWING -> REJECTED
```

状态含义：

| 状态 | 含义 |
| --- | --- |
| `DRAFT` | 草稿，可编辑 |
| `REVIEWING` | 审核中 |
| `NEED_REWRITE` | 需合规改写 |
| `REJECTED` | 审核拒绝 |
| `APPROVED` | 审核通过，可发布 |
| `PUBLISHED` | 已发布，可进入 C 端 |
| `OFFLINE` | 已下线 |

## 13. 后续实现顺序

建议按以下顺序继续推进：

1. `PromptModule`：Prompt 模板和版本管理。
2. `AssetModule`：素材上传、素材列表、素材描述和素材合规状态。
3. `AiGatewayModule`：LangChain + OpenAI 统一调用。
4. `ModerationModule`：规则库 + AI 审核，写入 `safety_reviews`。
5. `QualityModule`：AI 质量评分，写入 `quality_evaluations` 和维度明细。
6. `RewriteModule`：合规改写，写入 `rewrite_records` 并支持采纳生成新版本。
7. `RankingModule`：信息流、热点榜、质量榜、推荐榜。
8. `DistributionModule`：站内发布后的分发任务。

## 14. 后端验收

- 用户注册、登录、刷新 token、退出登录可用。
- JWT 鉴权可以正确保护 B 端写接口。
- 内容创建时能同时生成初始版本。
- 内容更新、回滚不会覆盖历史版本。
- 自动保存草稿能写入快照。
- 离线同步能识别服务端版本冲突。
- 发布前必须校验审核和评分结果。
- 所有接口返回统一响应结构。
- Swagger 和 Scalar 文档可访问。
- `pnpm --filter @xingliu/server build` 构建通过。
