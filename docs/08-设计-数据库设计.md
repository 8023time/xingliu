# 数据库设计

## 1. 原则

- PostgreSQL 是唯一业务数据库，不使用 Redis。
- 最终保留 14 张表：11 张核心表和 3 张独立加分项表。
- 草稿、正式版本和线上版本职责分离。
- 审核与评分必须关联具体内容版本。
- 榜单实时计算，不保存排名快照、热度分或推荐分。

## 2. 表划分

### 核心 11 表

| 表 | 职责 |
| --- | --- |
| `users` | 用户、认证状态和令牌版本 |
| `prompt_templates` | Prompt 主表 |
| `assets` | 素材及最新基础合规结果 |
| `contents` | 内容汇总、流程状态、当前版本和线上版本 |
| `content_versions` | 不可变正式版本 |
| `draft_snapshots` | 自动保存、恢复、离线同步和冲突 |
| `ai_tasks` | AI 生成、审核、评分和改写调用记录 |
| `safety_reviews` | 具体内容版本审核结果 |
| `quality_evaluations` | 具体内容版本质量评分 |
| `rewrite_records` | 合规改写候选及采纳状态 |
| `content_metrics` | 阅读等聚合指标 |

### 独立加分项 3 表

| 表 | 职责 |
| --- | --- |
| `content_interactions` | 点赞、分享等用户反馈明细 |
| `hot_topics` | 人工或外部 API 热点 |
| `distribution_tasks` | 外部平台分发任务 |

## 3. 内容与版本

`contents` 保存：

- 作者、内容类型、标题、摘要和封面。
- `current_version_id`：创作者当前编辑、审核或准备发布的正式版本。
- `published_version_id`：C 端当前公开展示的线上版本。
- 当前编辑版本流程状态、最新审核和评分汇总。
- 发布时间、下线时间和软删除时间。

`content_versions` 保存不可变正文版本。提交草稿、采纳改写和二次编辑审核都会创建新版本，禁止覆盖历史版本。

关键规则：

- 新建内容可以暂时没有正式版本。
- 提交审核时创建新版本并切换 `current_version_id`。
- 发布成功后才将 `published_version_id` 切换到当前版本。
- 二次编辑期间旧 `published_version_id` 继续公开。
- 下线清空 `published_version_id`。
- 公开详情、内容流和榜单只读取 `published_version_id`。

## 4. 草稿、审核与 AI

- `draft_snapshots` 使用 `base_version_id`、`client_revision` 和 `server_revision` 判断冲突；草稿不自动成为正式版本。
- `safety_reviews` 保存标准化结果、风险标签、命中片段、原因、请求 ID 和阿里云原始响应。
- `quality_evaluations.dimensions` 使用 JSON 保存多维评分。
- `rewrite_records` 保存候选改写；采纳后关联新建正式版本。
- `ai_tasks` 只保存任务元数据与脱敏摘要，不保存密钥和完整敏感正文。
- `assets` 只保存素材最新基础合规结果，不建设素材审核历史表。

## 5. 榜单数据

核心榜单根据 `contents.quality_score`、`content_metrics.view_count` 和 `contents.published_at` 实时计算。只有存在 `published_version_id` 且未删除、未下线的内容可参与榜单。

点赞、分享等 `content_interactions` 和附加指标只用于进阶智能排序。

## 6. 状态约束

```text
DRAFT → REVIEWING → NEED_REWRITE | REJECTED | APPROVED → PUBLISHED → OFFLINE
```

`status` 表示当前编辑版本流程状态，不单独决定内容是否公开。内容有 `published_version_id` 时，即使当前编辑版本处于 `DRAFT` 或 `REVIEWING`，旧线上版本仍可公开。

## 7. 已删除的冗余设计

不恢复用户角色、Prompt 版本、素材审核历史、审核规则配置、质量标准配置、评分维度明细、排名快照、热点内容关系、分发渠道和内容操作日志表。

## 8. 迁移

- `20260604160000_simplify_core_schema` 将旧结构收敛为 14 张表。
- 后续迁移增加 `published_version_id`，并使用当前已发布内容的 `current_version_id` 回填。
- 保留已有迁移历史；生产执行结构删除迁移前必须备份。
