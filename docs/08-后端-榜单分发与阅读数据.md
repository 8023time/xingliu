# 后端：榜单分发与阅读数据

## 设计目标

榜单系统用于展示 PDF 中的内容分发能力。MVP 不做复杂个性化推荐，优先支撑 C 端首页信息流、热点榜、爆文榜、推荐榜四个内容消费入口。主排序因子先采用内容质量分、阅读热度、发布时间；用户反馈作为进阶挑战预留。

## 分发入口

首页信息流：

- 综合推荐榜、新发布内容和近期热点，作为 C 端默认入口。
- 首屏优先读取缓存，保证移动端和桌面端打开速度。

## 榜单类型

热点榜：

- 看重近期热度和时间衰减。
- 适合展示“最近被关注”的内容。

爆文榜：

- 看重阅读量、点赞量、分享量等互动增长。
- 适合展示“互动表现好”的内容。

推荐榜：

- 看重质量分、安全分、阅读热度和新鲜度。
- 适合展示“值得推荐”的优质内容。

## 计分公式

互动分：

```text
engagementScore =
  viewCount * 1
  + likeCount * 5
  + shareCount * 12
```

时间衰减：

```text
timeDecay = 1 / pow(hoursSincePublish + 2, 1.2)
```

热点分：

```text
hotScore =
  engagementScore * timeDecay
  + qualityScore * 0.3
  + safetyScore * 0.2
```

推荐分：

```text
recommendScore =
  qualityScore * 0.45
  + safetyScore * 0.25
  + normalizedEngagement * 0.20
  + freshnessScore * 0.10
```

## Redis 缓存设计

```text
ranking:hot              热点榜内容 ID 有序集合
ranking:viral            爆文榜内容 ID 有序集合
ranking:recommend        推荐榜内容 ID 有序集合
feed:home                首页信息流内容 ID 有序集合或缓存列表
content:metrics:{id}     内容计数缓存
content:detail:{id}      内容详情短缓存
```

榜单读取流程：

1. 从 Redis Sorted Set 读取 contentId。
2. 批量查询 PostgreSQL 内容详情，或读取详情短缓存。
3. 返回 `items`、`nextCursor`、`hasMore` 和 `generatedAt`。
4. 前端使用 cursor 分页实现无限滚动。

## 阅读与互动事件

MVP 支持事件：

- 浏览。
- 点赞。
- 分享。

计数策略：

- 浏览按用户 + 内容 + 时间窗口去重，例如 30 分钟内重复浏览只计一次有效阅读。
- 点赞按用户 + 内容 + 类型唯一，重复点击表示取消。
- 高频计数先写 Redis，再批量落库。

进阶挑战可增加收藏、评论、举报、不感兴趣等反馈，并把反馈纳入动态加权排序。

## 排序刷新

MVP 刷新策略：

- 内容发布后立即计算一次分数。
- 用户阅读或互动后更新 Redis 计数。
- 每 1 到 5 分钟定时刷新榜单分数。
- Demo 阶段可在后端提供手动刷新接口，不单独创建独立脚本目录。

进阶策略：

- 记录每次刷新耗时、内容数和缓存命中率。
- 支持榜单权重配置化。
- 复杂度上来后再评估是否需要队列或独立任务服务。

## 榜单解释

榜单卡片或详情页可展示解释字段：

```json
{
  "rankReason": {
    "quality": "质量分 88，高于同类平均",
    "safety": "安全分 96，内容风险低",
    "engagement": "近 24 小时阅读和点赞增长较快",
    "freshness": "发布时间较新"
  }
}
```

答辩时可以讲清楚某篇内容为何排名靠前，体现推荐逻辑不是黑盒。

## 降级方案

- Redis 不可用：使用 PostgreSQL 索引字段排序，限制返回数量。
- 榜单刷新失败：使用上一轮缓存，并标记 `generatedAt`。
- 计数落库失败：保留 Redis 计数，下一轮后端任务重试。
- 高危内容复审失败：内容下线，避免继续分发高风险内容。
